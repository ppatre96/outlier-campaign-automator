<domain>
## Phase Boundary

**Problem:** New Smart Ramp requests sit in `submitted` state until someone manually runs `python main.py --ramp-id <id>`. With 9 ramps already in the queue and more arriving (5 over the past few weeks), this is the highest-leverage manual step left in the loop. The Smart Ramp client is already wired (`src/smart_ramp_client.py`) and the pipeline runs end-to-end on a single ramp; the missing piece is the trigger.

**Solution (Path A — polling, locked 2026-04-25):**
- 15-minute launchd-driven poll watches Smart Ramp for new + edited submitted ramps
- For each detected ramp, the pipeline runs end-to-end: Stage A/B (cohort discovery) → Stage 8 (creative generation) → Stage 9 (LinkedIn campaign creation as DRAFT)
- Both **InMail and Static** campaigns produced for every cohort (currently the pipeline picks one path; Phase 2.6 ensures both)
- LinkedIn `create_image_ad` is still 403-blocked → image saved locally with campaign-name-prefixed filename, ready for manual upload
- On success, single Slack DM to Pranav + Diego with all artifacts
- On 5 consecutive failures for the same ramp, escalation DM with traceback + manual-recovery command
- Edits handled by re-running with `_v2`/`_v3` suffix; old drafts tagged `superseded` in state file (left in DRAFT on LinkedIn for audit)

**Out of scope:**
- True webhook (Path B) — deferred until polling latency becomes a problem; would require Smart Ramp codebase changes + hosted receiver + queue
- Slack thread interactions (✅/🔁/⏸️ reactions) — deferred to V2 of this phase
- Auto-deleting superseded drafts — manual sweep
- Service-account Slack identity — DMs continue from Pranav's bot for now
- Updating live (non-DRAFT) campaigns — out of scope; Phase 2.6 only touches DRAFT artifacts

</domain>

<decisions>
## Implementation Decisions

### Architecture (Locked)

- **Path A polling, NOT webhook**. User locked this 2026-04-25. 15-min cadence is acceptable given current ramp arrival rate (~1/week). Upgrade to webhook is a future phase if rate jumps.
- **launchd, NOT crontab**. Phase 2.5 V2 just deduped crontab; we're not putting anything back. New plist `~/Library/LaunchAgents/com.outlier.smart-ramp-poller.plist` runs every 15 min via `StartInterval=900`. Separate from `com.outlier.weekly-reports.plist` so the two cadences stay decoupled.
- **State file is the source of truth.** `data/processed_ramps.json` tracks every ramp the poller has seen. Format:
  ```json
  {
    "ramps": {
      "GMR-0010": {
        "first_seen_at": "2026-04-25T10:00:00Z",
        "last_processed_at": "2026-04-25T10:15:00Z",
        "last_signature": "sha256:...",
        "consecutive_failures": 0,
        "last_failure_class": null,
        "version": 1,
        "campaign_groups": ["urn:li:sponsoredCampaignGroup:..."],
        "inmail_campaigns": ["urn:li:sponsoredCampaign:..."],
        "static_campaigns": ["urn:li:sponsoredCampaign:..."],
        "creative_paths": {
          "<cohort_id>_inmail_A": "data/ramp_creatives/GMR-0010/<cohort_id>_inmail_A.png",
          "<cohort_id>_static_A": "data/ramp_creatives/GMR-0010/<cohort_id>_static_A.png"
        },
        "superseded": false
      }
    }
  }
  ```
- **Content signature for edit detection.** `sha256(json.dumps(cohorts_sorted, sort_keys=True) + "\n" + summary + "\n" + updated_at)`. Pure content hash; updated_at alone is unreliable (Vercel sometimes refreshes it without real changes).
- **`_v2` versioning on edit.** When signature differs from `last_signature`: bump `version`, run pipeline producing new campaigns named `<existing_name>_v2`, write new entry, mark prior versions `superseded: true`. Don't delete old drafts on LinkedIn (manual sweep). Pranav can audit via `processed_ramps.json` history.
- **Test-ramp filter.** `requester_name.lower().contains("test")` → silently skip (don't process, don't DM, don't even count as "seen" — leave out of state file). Catches `Quintin Au Test` (GMR-0004) and any future test entries.
- **Concurrency guard.** `filelock` on `data/smart_ramp_poller.lock`. If a previous poll is still running (Redash can take 5+ min), the new launchd-fired poll acquires the lock with `timeout=5`; on timeout, log "previous poll still running" and exit 0. Same pattern as Phase 2.5 V2's weekly orchestrator.

### Pipeline Integration (Locked)

- **Both InMail and Static per cohort.** Today `main.py --ramp-id` runs one creative path per cohort row (decided by `_process_row` dispatch logic). Phase 2.6 needs both:
  - Plan 02 modifies `_process_row` (or adds a wrapper `_process_row_both()`) that runs the InMail path AND the Static-ad path for every cohort, producing two campaigns per cohort instead of one.
  - This requires understanding which arm `_process_row` currently picks — researcher must verify.
- **Image-attach 403 → local PNG.** Today the pipeline raises on `create_image_ad` 403. Phase 2.6 changes this to:
  - Catch the 403 in the campaign-creation path (Plan 02)
  - Save the generated PNG (already produced before the LinkedIn upload attempt) to `data/ramp_creatives/<ramp_id>/<cohort_id>_<inmail|static>_<angle>.png`
  - Embed the campaign name + URN in the filename or sidecar JSON for manual upload reference
  - Continue processing other cohorts (one cohort's image-attach failure does NOT abort the ramp)
  - Flag in `creative_paths` of state file so the Slack DM can list it
- **All campaigns DRAFT.** Locked per CLAUDE.md / project memory. No change.
- **Existing `--ramp-id` flow stays callable.** The poller invokes the pipeline programmatically; the manual `python main.py --ramp-id` CLI must keep working unchanged for ad-hoc runs.

### Slack Notifier (Locked)

- **Three notification targets per ramp**: solo DM to Pranav (`U095J930UEL`), solo DM to Diego (`U08AW9FCP27`), AND a post to channel `C0B0NBB986L` (added 2026-04-25). Three independent `chat.postMessage` calls (Slack doesn't support multi-recipient atomic DMs; two solo DMs + one channel post is the cleanest pattern). Identical message body for all three targets.

If neither holds, `chat.postMessage` returns `not_in_channel` — notifier catches that, logs a one-time warning, continues with the two DMs. Document the membership requirement in the launchd / setup README.

- **Reuse existing helpers.** `_post_to_slack()` from `scripts/post_weekly_reports.py` is the canonical sender (uses `SLACK_BOT_TOKEN`, falls back to webhook). The new notifier wraps it with per-recipient routing.
- **Vocabulary:** Every Slack-facing string honors CLAUDE.md. Notable: "draft" (not "campaign awaiting approval"), "review and activate" (not "approve"), "creative" (not "ad").

### Failure / Retry Semantics (Locked)

- **Every poll retries failed ramps.** No exponential backoff — 15-min interval is already long. Reset `consecutive_failures` to 0 on the first success.
- **Escalation threshold = 5.** Counter ≥ 5 → fire escalation DM, set `escalation_dm_sent: true` in state, stop retrying that ramp until `consecutive_failures` is reset to 0 in the state file (manual or by reply-thread-trigger in V2 of this phase).
- **Per-step isolation inside the poller.** If processing one ramp throws, log + record failure + continue with next ramp (don't abort the whole poll).
- **Transient errors counted same as permanent.** Redash timeouts, LiteLLM rate limits, LinkedIn 5xx — all increment `consecutive_failures`. Permanent errors (audience too small, missing required fields) tend to be obvious in the traceback summary on the escalation DM, so user can decide what to do.

### Scheduling (Locked)

- **Cron:** `~/Library/LaunchAgents/com.outlier.smart-ramp-poller.plist`, `StartInterval=900` (every 15 min), `RunAtLoad=true` so the first poll fires immediately on `launchctl load`.
- **Logs:** `logs/smart_ramp_poller/<yyyy-mm-dd>.log`, rotated daily by file naming. Same convention as Phase 2.5 V2 weekly orchestrator.
- **Working directory:** Must `cd /Users/pranavpatre/outlier-campaign-agent` before invocation OR pass an absolute working dir in the plist (`WorkingDirectory` key). Required for relative paths like `data/processed_ramps.json` to resolve.
- **load_dotenv first.** Before importing `config` or any module that reads env at import time. Same pattern as Phase 2.5 V2 weekly orchestrator (RESEARCH-V2 Pitfall 6).

### Claude's Discretion

- Whether to deduplicate the per-cohort LiteLLM copy generation between InMail and Static paths (they may need different copy — Static has tight char limits)
- Exact log format (structured JSON vs human-readable lines)
- Whether escalation DMs include the failed-cohort breakdown if SOME cohorts succeeded and others failed (recommended: include partial-success summary)
- Slack message threading vs single message — start with single message; thread if hits the 4000-char block limit
- Handling of `superseded` drafts during edit re-run: keep all version numbers in state file forever, or trim history older than N days (recommend keep all — small disk footprint)

<specifics>
## Specific Ideas

- The state file format includes a `superseded` boolean per version, but the latest version of a ramp is identified by `version` integer, not by a "current" pointer — simplifies edit handling because we can always look up `max(version)` for a given ramp_id.
- Slack DM should include a deep link to LinkedIn Campaign Manager (`https://www.linkedin.com/campaignmanager/accounts/510956407/campaigns`) so the user can jump straight to review.
- Image-local-save path embeds the campaign name in the filename so manual upload doesn't require cross-referencing a sidecar JSON. Format: `<cohort_id>_<inmail|static>_<angle>__<urlencoded_campaign_name>.png`. Example: `bn-in_static_A__agent_67e2f912_Coders_T2_India.png`.
- Diego may not have accepted the bot's DM yet. The notifier should catch the `not_in_channel` Slack error gracefully — fall back to webhook OR log a warning naming the user — and not break the whole poll.
- For ad-hoc debugging: a `--once --ramp-id <id> --dry-run` CLI flag on the poller script lets Pranav force-process a single ramp without waiting for the next 15-min tick or modifying state.
- Logs should include the content signature so we can debug "why did this ramp not re-trigger after edit?" by diffing signatures across log lines.

</specifics>
