## What Was Built

### `src/smart_ramp_notifier.py` (~228 LOC)

Public API:

- **`notify_success(ramp_record, result, version=1) -> dict`** — sends the success Slack message to all 3 targets in `config.SLACK_RAMP_NOTIFY_TARGETS`. Returns `{"user:U095J930UEL": True, "user:U08AW9FCP27": True, "channel:C0B0NBB986L": True}` (or False per failed target).
- **`notify_escalation(ramp_record, error_class, traceback_text) -> dict`** — same 3-target send shape; the message body contains the error class, last traceback line, manual-recovery command, and reset-counter Python snippet.
- **`build_success_message(ramp_id, project_name, requester_name, per_cohort, version=1) -> str`** — pure formatter; vocabulary-clean per CLAUDE.md (uses "draft", "review and activate", "creative"). Header bumps to `*Smart Ramp processed (v{version}): ...*` when `version > 1`.
- **`build_escalation_message(ramp_id, project_name, requester_name, error_class, traceback_text) -> str`** — pure formatter; includes `cd /Users/pranavpatre/outlier-campaign-agent`, `venv/bin/python3 main.py --ramp-id {ramp_id}`, and the reset-counter one-liner that resets BOTH `consecutive_failures=0` AND `escalation_dm_sent=False`.

Internals:

- **`_send_to_target(client, (kind, id), text) -> bool`** — dispatcher.
  - `kind == "user"`: `client.conversations_open(users=[target_id])["channel"]["id"]` → `client.chat_postMessage(channel=channel_id, text=text)`. Two-step path per RESEARCH §Q4 — differentiates `users_not_found` / `cannot_dm_bot` / `not_allowed_token_type` from generic `channel_not_found`.
  - `kind == "channel"`: direct `client.chat_postMessage(channel=target_id, text=text)`. No `conversations_open` prelude (channel ID is already canonical).
  - `SlackApiError` caught + logged with the error code + return `False` — caller iterates the next target. Per-target isolation: one failure NEVER blocks the others.
- **`_send_to_all_targets(text) -> dict`** — iterates `config.SLACK_RAMP_NOTIFY_TARGETS` (the 3-tuple list from Plan 01) and returns `{"user:UID": bool, ...}`. If `SLACK_BOT_TOKEN` is empty, returns all-False without calling Slack.

### `scripts/smart_ramp_poller.py` — STUB swap + notifier wiring

- **`run_ramp_pipeline(record, dry_run, version)` body REPLACED**: `from main import run_launch_for_ramp; return run_launch_for_ramp(record.id, modes=("inmail","static"), dry_run=dry_run)`. The `# STUB called` warning log line was removed. Surrounding orchestration (signature, escalation gate, atomic state write, action classification) is unchanged.
- **`process_ramp` extended**: now also returns `notify_kind` ∈ {"success", "escalation", None} and `record`. The transition logic is `prior_escalation = bool((prior or {}).get("escalation_dm_sent", False))` then `notify_kind = "escalation"` only when `not ok AND entry.escalation_dm_sent AND not prior_escalation AND not dry_run` — escalation fires exactly ONCE per threshold trip; subsequent polls (counter still high) return `notify_kind=None`.
- **`_drive_notifier(outcome, dry_run)` added**: drives the 3-target Slack send based on `outcome["notify_kind"]`. `notify_success` for success path; `notify_escalation` for the threshold transition. Both calls are wrapped in try/except so a Slack failure NEVER breaks the poll. In `--dry-run`, no Slack call is issued — logs `[DRY-RUN] would notify (kind=...)` instead.
- **`run_once` calls `_drive_notifier` after every `process_ramp`** — both in the normal poll loop and the `--ramp-id` force-process branch.

## Decisions Made

- **Vocabulary docstring rewrite.** The plan's automated vocabulary scan walks the source file looking for banned tokens (`compensation/project rate/job/role/...`) on non-comment lines. The original docstring listed the banned tokens by name (`NEVER emit banned tokens — compensation/project rate/job/role/...`) and tripped the scan because the listing line itself didn't have the magic skip word. Two fixes considered: (a) hyphenate the banned tokens; (b) reword the docstring to reference CLAUDE.md without inlining the list. Picked (b) because (a) embeds CLAUDE.md vocabulary in the source code, which means CLAUDE.md drift can silently desync from the docstring. The new docstring now reads `every user-facing string here uses approved Outlier vocabulary. The Don't-Say list (a.k.a. banned tokens) lives in CLAUDE.md and is never emitted at runtime — message bodies are exercised by the banned regex in tests/test_smart_ramp_notifier.py and the CI vocabulary scan.` (CLAUDE.md remains the source of truth.)
- **`_drive_notifier` extracted as a standalone helper** inside `scripts/smart_ramp_poller.py` (vs inline in `run_once`). The dual call sites (normal poll path + `--ramp-id` force-process branch) both need the same `notify_kind` dispatch + dry-run skip + per-call try/except. Extracting it puts that logic in one place and makes the two callers one-liners.
- **`process_ramp` returns `(record, notify_kind)` instead of calling Slack inline.** CONTEXT.md says the notifier sits OUTSIDE `process_ramp` so state writes stay isolated from Slack errors. The function now constructs the `notify_kind` decision (success vs escalation transition) and returns it; the caller (run_once) does the actual Slack call. Tests can assert `process_ramp`'s return shape without mocking Slack.
- **Integration test patches `scripts.smart_ramp_poller.SmartRampClient`** (not `src.smart_ramp_client.SmartRampClient`). The poller does `from src.smart_ramp_client import SmartRampClient, ...` at module load; the bound name lives in the poller's namespace. Patching the source module's symbol after import has no effect on the already-bound name. Same lesson as `tests/test_weekly_feedback_loop.py` from Phase 2.5 V2.
- **Integration test redirects `LOG_DIR` into `tmp_path`** so the poller's date-stamped log file (`logs/smart_ramp_poller/<yyyy-mm-dd>.log`) doesn't leak into the real `logs/` tree during test runs.

## Deviations from Plan

**Rule 1 — Bug fix (vocabulary scan).** The plan's verbatim docstring text emitted banned tokens as a literal list inside the module docstring (`NEVER emit banned tokens — compensation/project rate/job/role/position/interview/bonus/required/promote/assign/team.`). The plan's own vocabulary scan caught this on the listing lines (the skip word `banned` was on a different line). Net effect: the file passes the vocabulary scan, the runtime message bodies remain vocabulary-clean (verified by `test_dm_vocabulary`), and CLAUDE.md remains the single source of truth.
