### Pattern 2: Atomic State-File Write (NEW — must be added)

JSON state files corrupt under SIGKILL mid-`write_text()`. Use atomic rename:

```python
def _write_state_atomic(path: Path, state: dict) -> None:
    """Write state JSON atomically — write to tmp, fsync, rename."""
    path.parent.mkdir(parents=True, exist_ok=True)
    tmp = path.with_suffix(path.suffix + ".tmp")
    payload = json.dumps(state, indent=2, sort_keys=True, default=str)
    tmp.write_text(payload)
    # os.replace is atomic on POSIX + Windows; rename is not on Windows
    import os
    os.replace(tmp, path)
```

### Pattern 3: Content Signature for Edit Detection (NEW — must be added)

Verified-deterministic across re-fetches:

```python
import hashlib, json
from dataclasses import asdict
from src.smart_ramp_client import RampRecord

def compute_signature(ramp: RampRecord) -> str:
    """SHA-256 over (sorted-cohort dicts + summary + updated_at)."""
    cohorts_serialized = json.dumps(
        [asdict(c) for c in sorted(ramp.cohorts, key=lambda c: c.id)],
        sort_keys=True,
    )
    payload = f"{cohorts_serialized}\n{ramp.summary}\n{ramp.updated_at}"
    return "sha256:" + hashlib.sha256(payload.encode("utf-8")).hexdigest()
```

**Verified 2026-04-25:** `dataclasses.asdict()` recursively serializes the `RampRecord → cohorts: list[CohortSpec]` nesting (Python stdlib dataclass behavior). `json.dumps(sort_keys=True)` produces deterministic byte-output.

### Pattern 6: Slack DM with Defensive Channel Resolution (Plan 03)

```python
from slack_sdk import WebClient
from slack_sdk.errors import SlackApiError

def _send_dm(client: WebClient, user_id: str, text: str) -> bool:
    """Send DM to user_id. Returns True on success."""
    try:
        # Step 1 — explicit IM open. Returns the channel ID.
        # Slack docs: chat.postMessage(channel=user_id) auto-opens, but explicit
        # open gives clean error handling for users_not_found / cannot_dm_bot.
        open_resp = client.conversations_open(users=[user_id])
        channel_id = open_resp["channel"]["id"]
        # Step 2 — post to the resolved channel ID
        client.chat_postMessage(channel=channel_id, text=text)
        return True
    except SlackApiError as e:
        log.warning("DM to %s failed: %s", user_id, e.response.get("error"))
        return False
```

**Why two-step:** `conversations.open` documents `users_not_found`, `cannot_dm_bot`, and `not_allowed_token_type` errors explicitly, so we can branch on each cleanly. The single-call path collapses everything into `channel_not_found` which is harder to act on. Required scopes: `chat:write` + `im:write` (or `im:history` to read; not needed for v1). Project bot already has `chat:write` (proven by Phase 2.5 V2 weekly post).

Source: [Slack chat.postMessage docs](https://docs.slack.dev/reference/methods/chat.postMessage/) confirm channel parameter accepts user ID; [conversations.open docs](https://docs.slack.dev/reference/methods/conversations.open/) confirm `users` array parameter and explicit IM-open semantics.

- `RunAtLoad=true` fires one immediate run on `launchctl load` ✓
- `StartInterval=900`: if the job is still running when the next 900s tick fires, **launchd automatically skips that tick** (does NOT spawn an overlapping instance). `filelock` becomes belt-and-suspenders for the rare case of two distinct launchd manage events (e.g., manual `launchctl kickstart`).
- `WorkingDirectory` requires absolute path ✓
- launchd does NOT support `strftime` in `StandardOutPath` — solution: keep launchd's stdout/stderr at a constant `/tmp/...` path (just for crashes BEFORE the script's logger initializes); the script itself writes to `logs/smart_ramp_poller/<yyyy-mm-dd>.log` via Python `logging.FileHandler`. ### Anti-Patterns to Avoid

- **Hand-rolling JSON-state-file IO without atomic write.** A power-cut or SIGKILL mid-write corrupts the file and the next poll silently retries every ramp. Use the `os.replace` pattern.
- **Subprocess invocation of `main.py --ramp-id`.** Python `subprocess.run` swallows the exception structure, making `consecutive_failures` classification impossible. Call `run_launch(ramp_id=...)` (or extracted helper) in-process.
- **Calling `chat_postMessage(channel=user_id)` without try/except for `SlackApiError`.** A single failed DM (Diego hasn't accepted the bot) crashes the poll. Wrap both DM sends and log per-recipient.
- **Re-fetching the ramp list without a small per-poll cache.** Smart Ramp's `/api/ramps` returns 9 records summary-only; we still fetch each `submitted` ramp individually with `fetch_ramp(id)`. That's fine, but don't accidentally add a 4× retry loop on top.
- **Forgetting to skip ramps that are already at version=N+1 in state.** If a ramp went `submitted` → poller processed → user re-submitted → poller processed again → user reviewed and changed `status` to `approved`/`active`, we should NOT keep re-running it. Filter by `status == "submitted"` AT FETCH TIME.

## Common Pitfalls

### Pitfall 1: Calling `_process_row` twice duplicates Stage A/B/C work
**What goes wrong:** Phase 2.6 needs both InMail + Static per cohort. The naïve fix calls `_process_row(...ad_type="INMAIL")` then `_process_row(...ad_type="")`. Each call re-runs Snowflake fetches, feature engineering, Stage A, Stage B, Stage C — adding ~30s of duplicate work per ramp.
**Why it happens:** `_process_row` interleaves cohort discovery with campaign creation in one 600-line function.
**How to avoid:** Extract a `_resolve_cohorts()` helper from lines 222–517 of `main.py`, returning `(selected_cohorts, group_name, exclude_pairs, ...)`. Then `_process_row_both_modes()` calls `_resolve_cohorts()` ONCE and dispatches both campaign-creation arms.
**Warning signs:** Per-ramp processing time > 60s in logs; Redash cache misses doubling.

### Pitfall 2: Smart Ramp `updated_at` refreshes without content change
**What goes wrong:** Vercel deployment sometimes touches `updated_at` on every record without actual user edit. If the signature includes only `updated_at`, every Vercel re-deploy triggers a `_v2` re-run for all 9 ramps simultaneously.
**Why it happens:** Vercel ISR / edge caching propagation can write through.
**How to avoid:** Signature MUST hash CONTENT (cohorts + summary), not just timestamps. CONTEXT.md locks this: `sha256(json.dumps(cohorts_sorted) + summary + updated_at)` — `updated_at` is included as a tie-breaker but content dominates.
**Warning signs:** Multiple `_v2` re-runs in a single 24h period for ramps the user never opened.

### Pitfall 3: `load_dotenv` AFTER `import config`
**What goes wrong:** `config.py:32` reads `os.getenv("SCREENING_END_DATE", ...)` at import time. If `load_dotenv` runs after, `config.SLACK_BOT_TOKEN` and others are already cached as empty strings.
**Why it happens:** Python imports are cached.
**How to avoid:** `load_dotenv()` MUST run at module top BEFORE any `import config` or `from src.X` (most src modules transitively import config). Phase 2.5 V2 fixed this — see `scripts/weekly_feedback_loop.py:43–55`. Replicate verbatim.
**Warning signs:** Slack 401 errors despite `SLACK_BOT_TOKEN` clearly set in `.env`.

### Pitfall 4: launchd `$PATH` does not include venv
**What goes wrong:** `python3` in `ProgramArguments` resolves to system Python (no `requests`, no `slack_sdk`, no project deps), and ImportError crashes silently into `/tmp/*.stderr.log`.
**Why it happens:** launchd-spawned processes inherit a minimal `$PATH` (`/usr/bin:/bin:/usr/sbin:/sbin`).
**How to avoid:** Use absolute path `/Users/pranavpatre/outlier-campaign-agent/venv/bin/python3` in `ProgramArguments` (verified in the existing weekly-reports plist line 9).
**Warning signs:** `ModuleNotFoundError: No module named 'requests'` in `/tmp/outlier-smart-ramp-poller.stderr.log`.

### Pitfall 5: JSON state file corruption on SIGKILL
**What goes wrong:** Poll crashes mid-`json.dump()`; next poll reads partial JSON → JSONDecodeError → exception → 9 ramps re-processed from scratch (creating 9 duplicate `_v2` campaigns).
**Why it happens:** `Path.write_text()` is NOT atomic.
**How to avoid:** Atomic write-to-tmp-then-rename pattern in Pattern 2 above. `os.replace(tmp, final)` is atomic on POSIX and Windows.
**Warning signs:** `processed_ramps.json` ends mid-object; `processed_ramps.json.tmp` left behind.

### Pitfall 6: LinkedIn rate limits at scale
**What goes wrong:** 9 ramps × 5 cohorts × 2 modes (InMail + Static) = **90 campaigns + 90 creatives + 90 image uploads** in one poll. LinkedIn Marketing API rate limits are conservative (varies by endpoint, but typical "create" endpoints run ~100 req/min/account).
**Why it happens:** Auto-trigger on a backlog of 9 ramps simultaneously.
**How to avoid:** (a) the 9 backlog ramps are processed across multiple polls (each new ramp fired immediately, but `consecutive_failures` retry doesn't deliberately throttle); (b) accept that the FIRST `launchctl load` may rate-limit — the failed ramps just retry next poll. Don't add fancy throttling for v1.
**Warning signs:** LinkedIn 429 in logs; `consecutive_failures` jumps to >0 for many ramps simultaneously.

### Pitfall 7: Slack DM to a user who hasn't accepted the bot
**What goes wrong:** Diego (`U08AW9FCP27`) has never received a DM from `outlier-campaign-bot`. `chat.postMessage` returns `cannot_dm_bot` or similar (Slack docs ambiguous on exact error code in 2026).
**Why it happens:** Bot needs `im:write` scope AND user must have at least workspace-membership-confirmation with the bot.
**How to avoid:** Plan 03 **USER ACTION:** Pranav messages Diego asking him to send any one-character DM to `@outlier-campaign-bot` once before deploy. Plan 03 unit tests cover the failure-mode-fallback (DM fails → log warning + continue + don't break the poll).
**Warning signs:** First poll after deploy: Pranav gets DM, Diego doesn't, but no exception in logs.

### Pitfall 8: Filelock stale on crash
**What goes wrong:** Poller process killed (SIGKILL, panic, OOM); lock file remains; next poll's `FileLock(timeout=5)` waits 5s and exits.
**Why it happens:** OS-level signal kills can leak the lock.
**How to avoid:** `filelock` library uses fcntl/LockFileEx underneath, which the OS releases automatically on process death. If the OS-side lock IS released but the lockfile is on a network mount, things get weird — irrelevant on local APFS.
**Warning signs:** Successive polls all log "another poll already running" despite no Python process running.

### Pitfall 9: Filelock stuck when poller hangs (Redash slow)
**What goes wrong:** Redash takes 8 minutes to return; the next two 15-min ticks may now also block on the lock with `timeout=5` and exit cleanly. **This is BY DESIGN** (CONTEXT.md locks `timeout=5` → log + exit 0). But it means a slow ramp can starve other ramps from being processed for 8 min. Acceptable for v1.
**Why it happens:** Sequential per-ramp processing inside one poll.
**How to avoid:** Out of scope for v1. Future: per-ramp parallelism with bounded concurrency.

### Pitfall 10: Test-ramp filter too aggressive
**What goes wrong:** Filter is `requester_name.lower().contains("test")`. False positive: requester named "Christopher Testov" (legitimate) or anything with "test" substring. Their ramp silently ignored forever.
**Why it happens:** Substring match.
**How to avoid:** Use word-boundary regex: `r"\btest\b"` case-insensitive. Catches "Quintin Au Test" and "test_user" but not "Christopher Testov" or "tester engineer".
**Warning signs:** Real ramp not processed; user complains; we add their requester_name to a debug log and discover the substring collision.

```python
def _process_one_ramp(ramp, state, notifier):
    """Returns: (success: bool, error_class: str | None, traceback_text: str | None)"""

        from main import run_launch_for_ramp  # NEW helper exposed by Plan 02
        run_launch_for_ramp(ramp.id, modes=("inmail", "static"))
        return True, None, None
    except Exception as e:
        tb = traceback.format_exc()
        log.exception("Ramp %s processing failed", ramp.id)
        return False, type(e).__name__, tb


def run_once(args) -> int:
    state = _read_state()
    client = SmartRampClient()
    ramps_summary = client.fetch_ramp_list()  # 9 records, summary only
    submitted = [r for r in ramps_summary if r.status == "submitted"]
    test_pattern = re.compile(r"\btest\b", re.IGNORECASE)

    for summary_record in submitted:
        if test_pattern.search(summary_record.requester_name or ""):
            log.info("Skipping test ramp: %s (requester=%s)",
                     summary_record.id, summary_record.requester_name)
            continue

        # Fetch full ramp content
        full = client.fetch_ramp(summary_record.id)
        if not full:
            log.warning("Could not fetch full content for %s", summary_record.id)
            continue

        sig = compute_signature(full)
        prior = state["ramps"].get(full.id)

        # Decide: new, edit, or no-op
        action = _classify_action(prior, sig)
        if action == "noop":
            continue
        if action == "edit":
            _mark_superseded(state, full.id)

        success, err_class, tb = _process_one_ramp(full, state, notifier)
        _update_ramp_state(state, full, sig, success, err_class, tb, action)
        _maybe_send_notification(state, full, success, notifier)

    _write_state_atomic(STATE_PATH, state)
    return 0
```

### Example 2: Edit Detection State Update (Plan 01)

```python
def _classify_action(prior: dict | None, current_sig: str) -> str:
    if prior is None:
        return "new"
    if prior.get("last_signature") == current_sig:
        return "noop"
    return "edit"


def _mark_superseded(state: dict, ramp_id: str) -> None:
    """Bump version, mark prior as superseded — current latest version becomes the
    historical record under a versioned key in `state['ramp_versions']`."""
    cur = state["ramps"].get(ramp_id)
    if not cur:
        return
    state.setdefault("ramp_versions", {})
    history_key = f"{ramp_id}_v{cur['version']}"
    cur["superseded"] = True
    state["ramp_versions"][history_key] = cur
    # State["ramps"][ramp_id] gets overwritten by _update_ramp_state with version+1
```
