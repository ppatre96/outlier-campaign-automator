<domain>
## Phase Boundary

**Problem with v1:** v1 measures CTR + CPA at creative and cohort level, but CPA is "cost per application" — we learn nothing about whether the applicant signed up, passed screening, or activated as a paying contributor. A creative that generates many cheap applications but zero activations looks like a winner under v1 and a loser under V2. v1 also has no feedback from contributor sentiment (public reviews, support tickets) so copy doesn't know to address known pain points. And v1's cohort reanalysis is user-triggered via Slack reaction, so shifts in the underlying converter population go unnoticed until CPA degrades.

**V2 scope:** Extend the weekly feedback loop with three new data sources and wire them into one cron-scheduled run:

1. **Full-funnel conversion tracking** — For every creative × cohort, break down click → signup → screening-pass → activation. Identify the exact funnel stage where each underperforming cohort loses contributors and inject that diagnosis into the Slack alert.

2. **Sentiment miner** — Scrape Reddit, Trustpilot, Glassdoor, Apple/Play Store reviews, Outlier Community (Discourse), plus internal Zendesk/Intercom tickets. Extract themed issues via LLM. Produce `data/sentiment_callouts.json` that `ad-creative-brief-generator` reads as "address X / avoid Y" copy directives.

3. **Automatic ICP drift detection** — Snapshot Stage 1 ICP output weekly. Diff week-over-week on key feature distributions. When drift exceeds threshold, auto-trigger `outlier-data-analyst` reanalysis (no Slack reaction needed) so new inclusions/exclusions reach `campaign-manager` in the same weekly cycle.

4. **Cron orchestrator** — Single weekly job on Monday 9 AM IST wiring v1 `feedback_agent` + V2 funnel/sentiment/drift into one ordered run. Consolidated Slack report. Idempotent across reruns.

**Out of scope:**
- Real-time funnel dashboards (weekly cadence is the contract)
- Full NLP sentiment classifier training — use LLM-in-the-loop classification only
- Automatic copy regeneration — sentiment_miner *informs* the brief generator, it does not rewrite live creatives on its own
- A/B testing of sentiment-driven copy (that's still experiment_scientist_agent's job)
- Triggering more than one ICP reanalysis per week (rate-limit to one rerun/week/project to avoid churn)
- Statistical significance testing on drift (use simple threshold on KL divergence + minimum sample size)

</domain>

<decisions>
## Implementation Decisions

### Architecture (Locked)

- **Extend, don't replace**: v1 files (`src/feedback_agent.py`, `src/experiment_scientist_agent.py`, `src/reanalysis_loop.py`, `src/slack_alert_handler.py`) stay intact. V2 adds three new modules and one orchestrator script. Existing v1 tests must still pass.
- **Funnel extension placement**: The richer funnel logic already partially lives in `src/campaign_feedback_agent.py` (it queries `VIEW.APPLICATION_CONVERSION` for `applications` + `activations`). V2 *does not* fork into yet another agent. Plan 05 either (a) extends `src/feedback_agent.py` with a new `analyze_funnel_by_cohort()` method that queries the signup → screening → activation decomposition, or (b) wires `campaign_feedback_agent.run()` into `post_weekly_feedback_alert()` and deprecates the overlap. Planner decides which; recommend (a) to keep v1 contracts stable and add `campaign_feedback_agent` rollup in a later phase.
- **Sentiment miner is a standalone agent module**: `src/sentiment_miner.py` — one public `run(sources: list[str]) -> SentimentReport` entrypoint. Each source (Reddit, Trustpilot, Glassdoor, app stores, Discourse, Zendesk, Intercom) has its own fetcher function. Fetch failures are isolated (one source failing does not block the others).
- **ICP drift monitor is a standalone module**: `src/icp_drift_monitor.py` — weekly snapshot of `fetch_stage1_contributors(project_id)` output stored to `data/icp_snapshots/<project_id>/<yyyy-mm-dd>.parquet`. Drift score is KL divergence over binned categorical columns (skills, degree, field, job_title, experience_band) + absolute-mean shift on numeric columns (pay band if present). Threshold configured in `config.py` as `ICP_DRIFT_THRESHOLD` (default 0.15).
- **Cron orchestrator wires everything**: `scripts/weekly_feedback_loop.py` — sequentially calls: (1) existing v1 `post_weekly_feedback_alert()`, (2) V2 funnel analysis, (3) sentiment miner, (4) ICP drift check with conditional reanalysis, (5) consolidated Slack report. Each step is wrapped in try/except; failures in one step do not abort the others but are recorded in the Slack report.
- **Idempotency guard**: Orchestrator checks `data/weekly_feedback_loop_state.json` for `last_success_ts`. If run was successful within last 6 days, skip with no-op (protects against accidental double-cron or manual reruns the same week).

### Data Sources (Locked)

- **Full-funnel metrics** — Extend the existing Redash `VIEW.APPLICATION_CONVERSION` query pattern already used in `campaign_feedback_agent.py`. For the decomposition into screening-pass and activation, join against `PUBLIC.GROWTHRESUMESCREENINGRESULTS` + `VIEW.CONTRIBUTOR_EARLY_SUCCESS_FUNNEL` (both already referenced in `src/redash_db.py`). Cohort label comes from LinkedIn campaign `NAME` (same parsing as v1).
- **Sentiment sources — public**:
  - Reddit: subs `r/Outlier_AI`, `r/BeerMoney`, `r/WorkOnline`. Use public JSON endpoint (`https://www.reddit.com/r/<sub>/new.json?limit=100`) with a user-agent header; no auth required. Rate-limit to one call per sub per run.
  - Trustpilot: Firecrawl scrape of `https://www.trustpilot.com/review/outlier.ai`.
  - Glassdoor: Firecrawl scrape of `https://www.glassdoor.com/Reviews/Outlier-Reviews-E8756349.htm` (gracefully degrade if blocked).
  - Apple App Store / Google Play: app reviews pulled via their respective public endpoints (`itunes.apple.com/<country>/rss/customerreviews/...` for iOS; RSS feed equivalent or scraping-fallback for Play).
  - Outlier Community (Discourse): `https://community.outlier.ai/latest.json` if reachable; else Firecrawl scrape of `/latest` page.
- **Sentiment sources — internal**:
  - Zendesk: REST API via `ZENDESK_SUBDOMAIN` + `ZENDESK_EMAIL` + `ZENDESK_API_TOKEN` in `.env`. Endpoint: `/api/v2/search.json?query=type:ticket tags:contributor created>-7days`.
  - Intercom: REST API via `INTERCOM_ACCESS_TOKEN` in `.env`. Endpoint: `/conversations?created_since=<epoch_7d_ago>`. Use whichever Outlier CX uses; if neither, log and skip.
  - If neither credential is configured, skip internal sources with a warning — public sources still run.
- **Theme extraction**: Per-source fetched content is batched and sent to the LiteLLM gateway with a prompt that returns a JSON array of `{theme, evidence_quote, sentiment, source_url}`. Themes with >=3 supporting evidence quotes across any sources are promoted to `sentiment_callouts.json`. Lower-support themes are kept in raw dump but not surfaced to briefs.
- **ICP snapshots**: Weekly `fetch_stage1_contributors(project_id)` → saved as parquet (versioned per-week) in `data/icp_snapshots/<project_id>/`. Drift is the KL divergence between this week's feature distribution and the 4-week trailing median.

### Integration Points (Locked)

- **`ad-creative-brief-generator`** (agent): new required input `sentiment_callouts` (JSON path or inline dict). Brief generator must honor at least one "address" directive per run when the backlog has any. Existing brief generator prompt (`.claude/agents/ad-creative-brief-generator.md`) extended with a "Sentiment-Driven Copy Inputs" section.
- **`outlier-data-analyst`** (agent): `icp_drift_monitor` invokes it via the same path `reanalysis_loop.trigger_reanalysis()` uses today, but with `trigger_reason="icp_drift"` instead of Slack reaction. Rate-limit: at most one reanalysis per project per 7 days.
- **`campaign-manager`** (agent): no code change — consumes new cohorts from the same `data/cohort_queue.json` pattern v1 wrote to. If `reanalysis_loop` already writes there, V2 reuses; else V2 adds the write.
- **`feedback_agent`** (v1 class): gains `analyze_funnel_by_cohort()` and `identify_funnel_drop_stage()` methods. Existing methods untouched. `generate_slack_alert()` extended (backwards-compatible — new args default to empty).
- **Slack**: consolidated weekly post uses the existing `_post_to_slack()` helper in `scripts/post_weekly_reports.py`. New sections appended to the existing Monday post (not a second post), ordered: (1) v1 creative alerts, (2) funnel drops, (3) sentiment themes, (4) ICP drift.

### Scheduling (Locked)

- **Cron line**: `30 3 * * 1` (Monday 03:30 UTC = 09:00 IST). Exact same slot the existing Monday Slack cron uses — V2 orchestrator *replaces* the existing `post_weekly_reports.py` cron entry and calls it internally (preserving v1 behavior). Document this in the plan so we don't accidentally double-run.
- **Host**: Plan must document which host runs the cron. macOS laptop cron is unreliable (sleep/wake); if a server or GitHub Actions scheduled workflow is available, prefer that. If only the laptop is available, use `launchd` with `StartCalendarInterval` instead of `crontab` (survives reboots better on macOS).
- **Idempotency**: File-locked state in `data/weekly_feedback_loop_state.json`. Fields: `last_success_ts`, `last_failure_ts`, `last_failure_reason`, `runs_this_week`. Do not double-run if a successful run already occurred in the last 6 days.

### Thresholds (Locked — configurable in `config.py`)

- `ICP_DRIFT_THRESHOLD = 0.15` — KL divergence cutoff above which reanalysis auto-fires
- `ICP_DRIFT_MIN_ROWS = 200` — skip drift check if this week's Stage 1 output has fewer rows (noise floor)
- `SENTIMENT_THEME_MIN_EVIDENCE = 3` — theme needs this many supporting quotes to reach `sentiment_callouts.json`
- `SENTIMENT_LOOKBACK_DAYS = 7` — how far back to fetch reviews/tickets
- `FUNNEL_DROP_ALERT_THRESHOLD = 0.30` — if stage-to-stage conversion rate drops below this vs cohort baseline, flag the stage

### Language (Locked — per CLAUDE.md)

- Any Slack-facing copy uses Outlier vocabulary (see CLAUDE.md table). Key substitutions in this phase:
  - "screening" (not "interview" — already correct)
  - "payment" / "current tasking rate" (not "compensation" / "project rate")
  - "opportunity" / "task" (not "job" / "role")
  - "Outlier Community" (not "Discourse" in user-facing copy — use "Discourse" only in internal code/comments referring to the platform)
  - "progress" (not "performance" in alerts — "underperforming cohort" becomes "cohort with slowing progress")
  - "eligible to work on review-level tasks" (not "promote")
- `sentiment_miner` LLM prompt must strip/rewrite contributor quotes so the final JSON uses approved vocabulary when surfacing themes. Raw evidence quotes in `source_url` references are fine as-is.

### Claude's Discretion

- Exact KL-divergence binning strategy for numeric features (quantile vs fixed width) — planner picks
- Whether to use `trafilatura` or Firecrawl for web-scrape sources — planner picks based on reliability
- LiteLLM model choice for theme extraction (gpt-4.1-mini, claude-haiku-4.5, etc.) — planner picks lowest cost that gives reliable JSON
- Retry policy inside each sentiment fetcher — planner picks (1 retry + exponential backoff recommended)
- Parquet vs JSON for ICP snapshots — parquet recommended (smaller, faster diff) but JSON acceptable if dependency footprint is a concern
- Where to surface raw sentiment dump (Slack thread vs Drive link) — planner picks based on length

</decisions>

<specifics>
## Specific Ideas

- Funnel decomposition should use rates (%), not absolute counts, so low-volume cohorts don't get spuriously flagged. Absolute counts still displayed alongside for context.
- For the Slack weekly report, prefer one long consolidated message over multiple posts. Channel threading is acceptable for raw dumps but the top-line must be readable in a single glance.
- Reanalysis auto-trigger should be paused for the current week if that project already had a reanalysis in the last 7 days (log and skip).
- ICP snapshots should be kept indefinitely — the trailing 4-week median comparison will always need 4 weeks of history.
- Sentiment miner should NOT scrape or fetch user-level PII (names, emails). Only public review text and ticket metadata (theme, timestamp). Ticket bodies can be summarized and discarded; do not persist raw ticket text.
- Cron log output should go to `logs/weekly_feedback_loop/<yyyy-mm-dd>.log` for postmortem review.
- On failure, the orchestrator should still post a minimal Slack message saying "weekly feedback loop failed at step X" with the error — silent failure is worse than loud failure for a cron job.

</specifics>

<deferred>
## Deferred Ideas

- **Per-sub Reddit sentiment weights**: treat `r/Outlier_AI` signal as higher-weight than `r/BeerMoney` (the former is Outlier-specific, latter is cross-platform). Defer: start with equal weight, tune if needed.
- **Causal attribution from sentiment to CPA change**: correlating "new support theme emerged" with "CPA moved" would need a longer time series. Defer to a later phase once 8+ weeks of history exists.
- **Automatic copy regeneration**: sentiment_miner could trigger `outlier-copy-writer` to auto-regenerate headlines. Deferred — keep human-in-the-loop (brief generator gets the directive, human reviews, experiment_scientist_agent promotes to test).
- **Multi-project ICP drift correlation**: detecting "all our projects are seeing the same ICP drift" would point to platform-wide shifts. Useful but needs multiple active projects first.
- **Statistical significance testing on funnel drops**: right now we use simple threshold. Chi-square or bootstrap CI deferred until we have enough volume per cohort.

</deferred>
