# Phase 2.5 Plan 05: Full-Funnel Conversion Tracking Summary

**One-liner:** Extends shipped v1 FeedbackAgent with click → signup → screening-pass → activation funnel decomposition (FEED-15) and earliest-stage drop diagnosis (FEED-16) injected into the existing weekly Slack alert via a backwards-compatible kwarg.

## What Was Built

V2 extension to the shipped Phase 2.5 v1 FeedbackAgent. Three new public APIs and one new SQL constant, all additive — no v1 contracts broken:

1. **`RedashClient.query_funnel_metrics(days_back, account_id=None)`** — adapts the proven `_METRICS_SQL` pattern from `src/campaign_feedback_agent.py:60-127` (LinkedIn creatives ↔ APPLICATION_CONVERSION) and adds one `LEFT JOIN PUBLIC.GROWTHRESUMESCREENINGRESULTS g ON ac.EMAIL = g.CANDIDATE_EMAIL`. Returns a 12-column DataFrame: cohort_name, creative_id, impressions, clicks, spend, applications, screening_passes, activations, ctr, click_to_signup, signup_to_screen, screen_to_activate. UTM filters are `UTM_SOURCE ILIKE '%linkedin%'` + `UTM_MEDIUM = 'paid'`. PASS filter is case-insensitive (`UPPER(g.RESULT) = 'PASS'`). `AD_ID` is coerced via `TRY_TO_NUMBER` because `APPLICATION_CONVERSION.AD_ID` is text.

2. **`FeedbackAgent.analyze_funnel_by_cohort(days_back)`** — thin wrapper around `query_funnel_metrics` returning `list[dict]` keyed by `(cohort_name, creative_id)`. Empty list on empty Redash response.

3. **`FeedbackAgent.identify_funnel_drop_stage(funnel_rows)`** — for each cohort, computes the cohort-median rate at each of four stages (top_of_funnel/ctr, signup/click_to_signup, screening/signup_to_screen, activation/screen_to_activate) and picks the EARLIEST stage whose worst observed rate is `<= median * (1 - FUNNEL_DROP_ALERT_THRESHOLD)`. Returns `{cohort_name -> {drop_stage, drop_rate, baseline_rate, delta_pct}}`. `drop_stage` is one of `top_of_funnel | signup | screening | activation | none`. Cohorts with no qualifying drop get `drop_stage = "none"` and `None` rates.

4. **`FeedbackAgent.generate_slack_alert` extended** — added optional `funnel_diagnosis: dict | None = None` kwarg (defaults to None). When provided AND at least one cohort has a non-`none` drop_stage, appends a "Funnel Drop Diagnosis:" section between the existing "Top Underperforming Cohorts:" and "Creative Insights:" blocks. Section uses neutral wording per CLAUDE.md vocabulary — "funnel drop", "stage", "rate", "baseline", "delta". No banned tokens. Cohorts with `drop_stage = "none"` are silently skipped from the diagnosis section.

5. **`config.FUNNEL_DROP_ALERT_THRESHOLD`** — new module constant, default `0.30`, configurable via `FUNNEL_DROP_ALERT_THRESHOLD` env var.

## How to Use

```python
from src.feedback_agent import FeedbackAgent
from src.redash_db import RedashClient

agent = FeedbackAgent(RedashClient())

# v1 flow (unchanged) — still works exactly as before
underperformers = agent.identify_underperforming_cohorts(days_back=7)
hypotheses      = agent.analyze_creative_performance(days_back=7)

# V2 funnel flow
funnel_rows = agent.analyze_funnel_by_cohort(days_back=7)
diagnosis   = agent.identify_funnel_drop_stage(funnel_rows)

# Inject diagnosis into existing weekly Slack alert (backwards-compatible)
alert = agent.generate_slack_alert(
    underperformers,
    hypotheses,
    funnel_diagnosis=diagnosis,   # OPTIONAL — omit for v1 behavior
)
```

The kwarg defaults to None, so existing v1 callers (e.g. `scripts/post_weekly_reports.py`) continue to work unchanged.

## Key Implementation Details

- **SQL pattern source:** `src/campaign_feedback_agent.py:60-127` already joined `AD_ANALYTICS_BY_CREATIVE` ↔ `APPLICATION_CONVERSION` for applications + activations per `creative_id`. The V2 funnel SQL is a near-verbatim adaptation that adds one `LEFT JOIN PUBLIC.GROWTHRESUMESCREENINGRESULTS g ON ac.EMAIL = g.CANDIDATE_EMAIL` and a `COUNT(DISTINCT CASE WHEN UPPER(g.RESULT) = 'PASS' THEN ac.EMAIL END)` to compute `screening_passes` per creative. Rates are computed as `ratio_a/ratio_b` with `NULLIF(divisor, 0)` to safely handle zero-volume rows.

- **Avoiding row multiplication on the screening join:** `COUNT(DISTINCT ac.EMAIL)` (not `COUNT(*)` and not `COUNT(DISTINCT g.SCREENING_ID)`) so a contributor with multiple screening attempts is counted exactly once.

- **Median baseline:** the cohort-median is used as the per-stage baseline (not mean) — one outlier creative dragging the average down won't poison the diagnosis.

- **Earliest-stage-wins:** when multiple stages drop, the earliest is reported. This is causally correct — fixing a top-of-funnel CTR problem is a precondition for diagnosing a downstream signup/screening drop.

- **Vocabulary compliance:** the new Slack section uses "funnel drop at {stage} stage", "rate", "baseline", "delta". No "compensation", "interview", "project rate", " job ", " role " anywhere. The existing v1 Slack alert section was unchanged, so its wording is unaffected.
