<domain>
## Phase Boundary

**Problem:** Current pipeline generates creatives and runs campaigns but has no feedback mechanism to identify which creatives/cohorts work, why, or what to test next.

**Solution:** Two new specialist agents enable continuous optimization:
1. **feedback_agent** — Analyzes creative and cohort performance weekly, generates hypotheses, alerts on underperformers
2. **experiment_scientist_agent** — Accumulates signals (feedback + competitor intel), maintains experiment backlog, drives A/B testing

**Scope:**
- Creative performance loop: Identify best/worst performing headlines/photos per cohort, generate hypotheses
- Cohort performance loop: Detect underperforming cohorts (CPA > baseline + 2σ or CTR trending ↓), post weekly Slack alerts
- Experiment design: Use feedback + competitor data to decide what to test, feed test directives to creative generator
- Reanalysis loop: User triggers cohort reanalysis on fresh screening data, new cohorts discovered and fed back to pipeline

**Out of scope:**
- Automated cohort pause (user must confirm via Slack)
- Statistical significance testing (MVP uses thresholds)
- Full A/B test harness (we run campaigns, not test/control groups)
- Causal inference beyond correlation hypotheses

</domain>

<decisions>
## Implementation Decisions

### Architecture Decisions (Locked)

- **feedback_agent scope split:** Separate concerns into creative performance and cohort performance. Creative analysis drives experiment hypotheses. Cohort analysis drives user alerts.
- **experiment_scientist_agent as consolidator:** Single agent accumulates feedback + competitor intel + new screening data, decides what to test. Prevents scattered decision-making.
- **80/20 baseline vs. test rule:** 80% of creatives use proven baseline angles, 20% use test variants from experiment backlog. Balances learning and stability.
- **Weekly cycle timing:** Align with existing Monday 9 AM IST Slack reporting window (Phase 2). Single coherent feedback loop, not scattered ad-hoc tests.
- **Memory persistence:** Experiment backlog, hypothesis results, and test tracking stored in memory system. Survives process restarts.

### Data Architecture (Locked)

- **Performance source:** `VIEW.LINKEDIN_CREATIVE_COSTS` for creative-level metrics (impressions, clicks, spend, CPA by creative)
- **Cohort source:** `VIEW.APPLICATION_CONVERSION` for cohort-level metrics (CTR, CPA, spend trend)
- **Competitor source:** `data/competitor_intel/latest.json` from competitor_bot (weekly refresh)
- **Experiment metadata:** Creative metadata flags experiment status (`experiment: true/false`, `test_variant: angle/photo_subject/targeting`)
- **Cohort reanalysis trigger:** User Slack reaction (👍 pause / 🧪 test new angles) → triggers outlier_data_analyst

### Integration Points (Locked)

- **ad-creative-brief-generator:** Receives test directive from experiment_scientist_agent. 20% of briefs incorporate test angle/photo_subject.
- **outlier_data_analyst:** 
  - Rerun on cohort underperformance (triggered by Slack reaction)
  - New screening data (weekly) → discover new cohorts
  - Results fed back to campaign-manager for next run
- **campaign-manager:** Accepts new cohort definitions from Stage A reanalysis, schedules campaigns for newly discovered cohorts
- **Slack:** Weekly alert with top 3 underperforming cohorts + recommendation (PAUSE / TEST_NEW_ANGLES)

### Claude's Discretion

- **Performance threshold tuning:** What CPA/CTR delta triggers an "underperforming" alert (MVP: 2σ above baseline, or 10% CTR decline)
- **Hypothesis generation algorithm:** How feedback_agent generates "why is angle X better for cohort Y?" hypotheses (rule-based pattern matching vs. LLM-generated)
- **Experiment priority scoring:** How experiment_scientist_agent ranks which hypotheses to test first (impact × confidence × feasibility)
- **Test duration:** How long to run test variant before declaring "winner" (MVP: run 1 week, evaluate next Monday)
- **Failure handling:** What to do if feedback_agent or experiment_scientist_agent crashes (fallback to manual review, keep pipeline running)

</decisions>

<specifics>
## Specific Requirements from PRD

### Creative Performance Loop
- **FEED-01:** feedback_agent reads creative performance from `VIEW.LINKEDIN_CREATIVE_COSTS` (cohort × angle × metric)
- **FEED-02:** feedback_agent generates hypothesis JSON `{angle, photo_subject, reason_hypothesis, expected_impact}`
- **FEED-03:** experiment_scientist_agent receives hypothesis JSON + competitor_bot JSON
- **FEED-04:** experiment_scientist_agent outputs test decision: `{cohort, angle, photo_subject, priority}`

### Cohort Performance Loop
- **FEED-05:** feedback_agent reads cohort-level metrics from `VIEW.APPLICATION_CONVERSION` (cohort × CPA, CTR, spend, trend)
- **FEED-06:** feedback_agent identifies underperforming cohorts (CPA > baseline + 2σ OR CTR trending down)
- **FEED-07:** feedback_agent posts weekly Slack alert with underperforming cohort list + recommendation
- **FEED-08:** User acknowledges Slack alert (👍 = pause cohort, 🧪 = test new angles)

### Reanalysis Loop
- **FEED-09:** On user action (pause), outlier_data_analyst reruns Stage A with fresh screening data
- **FEED-10:** New cohorts discovered from fresh data fed back to campaign-manager
- **FEED-11:** campaign-manager schedules new campaigns for newly discovered cohorts

### Experiment Backlog & Tracking
- **FEED-12:** experiment_scientist_agent maintains experiment backlog (priority queue of hypotheses)
- **FEED-13:** ad-creative-brief-generator checks backlog; occasionally uses test variant instead of baseline
- **FEED-14:** After test runs, results fed back to feedback_agent to validate hypothesis

### Weekly Cycle
1. feedback_agent analyzes creative & cohort performance, generates hypotheses, posts Slack alert
2. experiment_scientist_agent reads feedback, competitor intel, decides what to test
3. Per-run: ad-creative-brief-generator checks backlog (20% test, 80% baseline)
4. User reacts to Slack alert to pause or test
5. outlier_data_analyst reruns on fresh screening data
6. New cohorts fed back to campaign-manager

</specifics>

- Multi-arm bandit for test allocation (Phase 3+) — MVP uses fixed 80/20 split
- Causal inference engine (Phase 3+) — MVP uses correlation hypotheses only
- Personalized angle recommendations per individual profile (Phase 3+) — MVP uses cohort-level aggregation
