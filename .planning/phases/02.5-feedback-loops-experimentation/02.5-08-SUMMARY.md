## Key Implementation Details

- **`load_dotenv()` BEFORE `import config`** — module-top order is critical. `config.py` reads env vars at import time (Pitfall 6); reversing the order would silently zero out `ZENDESK_API_TOKEN`, `INTERCOM_ACCESS_TOKEN`, etc.

- **`filelock.FileLock` wraps `run_once`** — 10-second acquisition timeout. Concurrent invocations exit cleanly with code 0 and "another instance running" log line (Pitfall 5 fix). The state file write happens INSIDE the lock so reads and writes are mutually exclusive.

- **6-day SKIP_WINDOW with strict `<` comparison** — `datetime.now(utc) - last_ts < timedelta(days=6)` returns True only when STRICTLY less than 6 days have elapsed. Boundary case (exactly 6 days, or in the future via clock skew) falls through to run. `--force` bypasses unconditionally. Malformed `last_success_ts` strings are caught and treated as "never ran".

- **Step isolation contract** — every step function returns `dict` with `"ok": bool` and on failure `"error": "<ExceptionClass>: <message>"`. The body of every step is wrapped in `try/except Exception` so a Redash 5xx in funnel cannot abort sentiment or drift. Even `_step_drift` has a try/except around the per-project loop AND a try/except around the outer `from src import icp_drift_monitor as idm` import.

- **Loud failure** — `run_once` ALWAYS attempts to post to Slack. On partial/total failure, `_build_failure_message` is used (minimal: lists each failed step and its error class). On full success, `_build_consolidated_message` builds the four-section rich message. The Slack post itself is wrapped in a final try/except so even Slack going down doesn't escalate to a process crash.

- **Dry-run safety** — `--dry-run` flows through to every step:
  - `_step_v1` short-circuits and returns `{"ok": True, "dry_run": True}` without calling `post_weekly_reports.main()`.
  - `_step_funnel` and `_step_sentiment` ignore `dry_run` (they do not produce side-effects on their own — they just compute data).
  - `_step_drift` calls `idm.snapshot(pid)` + `idm.compute_drift(pid)` directly, BYPASSING `idm.check_and_trigger(pid)` so the outlier-data-analyst orchestrator is NEVER invoked.
  - `run_once` checks the dry_run flag at the consolidated-Slack point and logs the message via `log.info` instead of calling `_post_to_slack`.

- **Section order is locked** — Creative Progress Alerts (v1) -> Funnel Drop Diagnosis -> Sentiment Themes -> ICP Drift. This matches CONTEXT-V2's locked decision and the four-section contract behind FEED-23.

- **Vocabulary cleanliness** — every Slack-facing string in `_build_consolidated_message`, `_build_failure_message`, and the four `_format_*_section` helpers uses approved Outlier vocabulary. Banned tokens (`compensation`, `project rate`, `interview`, `bonus`, `assign`, `job`, `role`, `position`, `promote`, `required`) NEVER appear in any emitted line. Verified by:
  1. The plan's runtime regex check (`grep -E '\\b<banned>\\b'` excluding NEVER/banned/comment markers) — passes with 0 hits.
  2. The unit test `test_slack_vocabulary` which compiles a `re.compile(...)` over the same banlist and runs it against a synthesized message AND a synthesized failure message.

- **In-process v1 call** — Step A imports `scripts.post_weekly_reports` and calls `post_weekly_reports.main()` directly. This keeps v1 exceptions (e.g., Redash timeout, Slack auth error) catchable in the orchestrator's try/except. We also catch `SystemExit` (in case the v1 script ever calls `sys.exit(0)` internally) and treat it as a successful run.

- **Active projects resolution** — `_active_projects()` checks three sources in order: (1) `data/active_projects.json` (JSON list of strings), (2) `OUTLIER_TRACKING_PROJECT_ID` env var, (3) empty list with logged warning. The empty case does NOT raise — drift step short-circuits with `{"ok": True, "note": "no_active_projects"}` and other steps still run.

- **Logging** — `_setup_logging()` configures both a date-stamped FileHandler at `logs/weekly_feedback_loop/<yyyy-mm-dd>.log` AND a stdout StreamHandler. Old handlers are cleared on each invocation so re-running in the same Python process (e.g. tests) doesn't double-log.
